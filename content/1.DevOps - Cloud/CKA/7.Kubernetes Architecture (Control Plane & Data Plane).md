# ☸️ Day 7: Kubernetes Architecture (Control Plane & Data Plane)

> [!summary] Video Overview
> This note dives deep into **Kubernetes architecture**, covering:
> - Pods & Deployments (foundation)
> - Control Plane vs Data Plane
> - Core components (API Server, Scheduler, etc.)
> - End-to-end deployment workflow
>
> 📌 This is one of the MOST IMPORTANT topics for CKA.

---

## 📦 Concept 1: What is a Pod?

> [!important]
> A Pod is the **smallest deployable unit in Kubernetes**

### 🧠 Definition
- Pod = Wrapper around container(s)
- Can contain:
  - 1 container (most common)
  - Multiple containers (advanced use cases)

---

### 📦 Pod = Container Box

- Docker → Container  
- Kubernetes → Pod (contains container)

---

### 🔗 Multi-Container Pods (Sidecar Pattern)

> [!info]
> Multiple containers inside a pod follow **Sidecar Pattern**

### Common Use Cases:
- 📜 Logging → collect logs
- 📊 Monitoring → send metrics
- 🌐 Proxy → outbound requests
- 🔁 Reverse Proxy → handle incoming traffic

---

### 🌐 Shared Resources in Pod

> [!important]
> Containers inside a pod share:

- 🌐 **Network**
  - Same IP
  - Access via `localhost`
- 💾 **Storage**
  - Shared volumes

---

## 🚀 Concept 2: What is a Deployment?

> [!important]
> Deployment ensures your app is always running

### 🔑 Why not just Pod?
- If pod dies → no recovery ❌

---

### ✅ Deployment Features:

#### 📈 1. Replica Management
- Define number of pods:
  - Example: `replicas = 10`
- Kubernetes ensures 10 always run

---

#### 🔄 2. Rolling Updates & Rollbacks
- Update without downtime
- Rollback if failure

---

#### 📜 3. Declarative Configuration

> [!quote]
> You define "desired state" → Kubernetes maintains it

---

## 🧠 Concept 3: Kubernetes Architecture Overview

> [!example]
> Cluster = Control Plane + Data Plane

---

### 🧠 Control Plane (Brain 🧠)

> [!important]
> Manages entire cluster

- Must be **Highly Available**
- Responsible for:
  - Decisions
  - Scheduling
  - State management

---

### ⚙️ Data Plane (Workers)

> [!important]
> Runs actual applications (Pods)

- Contains:
  - Worker Nodes
  - Containers

---

## 🧩 Concept 4: Control Plane Components

---

### 🌐 API Server

> [!important]
> Entry point to Kubernetes

- All requests go through API Server
- Handles:
  - Authentication
  - Authorization
  - Validation

---

### 🗄️ etcd

> [!important]
> Key-value database

- Stores:
  - Cluster state
  - Configurations
- Must be:
  - Backed up
  - Highly available

---

### ⚙️ Controller Manager

> [!important]
> Maintains desired state

- Runs multiple controllers:
  - Deployment controller
  - ReplicaSet controller

---

### 📍 Scheduler

> [!important]
> Assigns pods to nodes

- Based on:
  - CPU / Memory
  - Rules (e.g., GPU required)

---

### ☁️ Cloud Controller Manager

> [!info]
> Connects Kubernetes with cloud providers

- AWS, Azure, GCP integrations

---

## 🧩 Concept 5: Data Plane Components

---

### 🔄 Kubelet

> [!important]
> Node agent

- Runs on every node
- Responsibilities:
  - Start containers
  - Monitor health
  - Talk to API Server

---

### ⚙️ Container Runtime

> [!info]
> Runs containers

- Examples:
  - containerd
  - CRI-O

---

### 🌐 kube-proxy

> [!important]
> Network traffic manager

- Handles:
  - Service → Pod routing
  - Load balancing
  - Health checks

---

## 🌐 Concept 6: Kubernetes Service

> [!important]
> Logical abstraction for pods

### ❌ Problem:
- Pods have dynamic IPs

### ✅ Solution:
- Use **Service**
- Traffic goes:
  ```
  Pod → Service → Target Pod
  ```

---

## 🔌 Concept 7: CNI (Container Network Interface)

> [!important]
> Core networking system in Kubernetes

### Responsibilities:
- Assign IPs to pods
- Pod-to-Pod communication
- Network policies
- IP management

---

### ⚡ Advanced Insight

> [!tip]
> Some CNI plugins can replace kube-proxy

Examples:
- Calico
- Cilium

---

## 🧠 Concept 8: Deployment Workflow (VERY IMPORTANT)

> [!example] End-to-End Flow

### Step 1: User Command
```bash
kubectl apply -f app.yaml
```

---

### Step 2: kubectl
- Validates YAML
- Sends request to API Server

---

### Step 3: API Server
- Authenticates user
- Validates schema
- Stores in etcd

---

### Step 4: Controller Manager
- Detects mismatch
- Creates:
  - Deployment → ReplicaSet → Pods

---

### Step 5: Scheduler
- Assigns nodes to pods

---

### Step 6: Kubelet
- Pulls image
- Starts containers

---

### 🔁 Final Flow Summary

> [!quote]
> kubectl → API Server → etcd → Controller → Scheduler → Kubelet → Container Runtime

---

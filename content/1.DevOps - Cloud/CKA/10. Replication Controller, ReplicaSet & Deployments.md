# ☸️ Day 10: Replication Controller, ReplicaSet & Deployments

> [!summary] Video Overview
> This note covers how Kubernetes manages workloads using:
> - Replication Controllers  
> - ReplicaSets  
> - Deployments  
> - Selectors (Equality & Set-based)  
>
> 📌 Core idea: **Maintain desired number of Pods automatically**

---

## 📦 Concept 1: Why Controllers?

> [!important]
> Pods alone are NOT reliable

### ❌ Problem:
- Pod crashes → no auto-restart
- No scaling
- No management

---

### ✅ Solution:
- Use **Controllers**
- Ensure:
  - High availability  
  - Self-healing  
  - Scaling  

---

## 🔁 Concept 2: Replication Controller (RC)

> [!info]
> First method to manage multiple pods

---

### 🧠 What it does:
- Maintains desired number of pods
- If pod dies → creates new one

---

### 📄 RC YAML Structure

```yaml
apiVersion: v1
kind: ReplicationController

metadata:
  name: nginx-rc

spec:
  replicas: 3

  selector:
    app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
        - name: nginx-container
          image: nginx
```

---

### 🎯 Key Concept: Labels & Selectors

> [!important]
> RC works ONLY based on labels

- Selector:
```yaml
app: nginx
```

- It manages:
  - Pods having label `app=nginx`

---

### ⚠️ Important Behavior

> [!warning]

- Existing pods with matching labels are also managed
- RC may NOT create new pods if matching ones already exist

---

## ⚙️ Useful Commands (RC)

```bash
kubectl get rc
kubectl describe rc nginx-rc
kubectl scale rc nginx-rc --replicas=4
kubectl set image rc nginx-rc nginx-container=nginx:1.22
kubectl edit rc nginx-rc
```

---

## 🚀 Concept 3: ReplicaSet (RS)

> [!important]
> Modern replacement of Replication Controller

---

### 🧠 Key Difference:
- Supports **Set-based selectors** (more powerful)

---

### 📄 RS YAML

```yaml
apiVersion: apps/v1
kind: ReplicaSet

metadata:
  name: nginx-rs

spec:
  replicas: 3

  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
        - name: nginx-container
          image: nginx
```

---

## 🔍 Concept 4: Types of Selectors

---

### 🔹 1. Equality-Based Selector

> [!example]

```yaml
matchLabels:
  app: nginx
```

- Exact match required

---

### 🔹 2. Set-Based Selector

> [!example]

```yaml
matchExpressions:
  - key: app
    operator: In
    values:
      - nginx
      - apache
```

---

### 🔑 Operators:

| Operator       | Meaning                         |
|----------------|---------------------------------|
| In             | Value must be in list           |
| NotIn          | Value must NOT be in list       |
| Exists         | Key must exist                  |
| DoesNotExist   | Key must NOT exist              |

---

### 💡 Example (Advanced)

```yaml
matchExpressions:
  - key: app
    operator: In
    values: [nginx, apache]

  - key: environment
    operator: NotIn
    values: [development]

  - key: tier
    operator: Exists

  - key: debug
    operator: DoesNotExist
```

---

> [!important]
> ALL conditions must be satisfied (AND logic)

---

## ⚠️ RC vs RS

| Feature              | RC              | RS              |
|---------------------|-----------------|-----------------|
| Equality Selector   | ✅ Yes           | ✅ Yes           |
| Set-Based Selector  | ❌ No            | ✅ Yes           |
| Usage Today         | ❌ Deprecated    | ⚠️ Rarely used   |

## 🚀 Concept 5: Deployment (MOST IMPORTANT)

> [!success]
> Deployment = Highest-level controller (used in real world)

---

### 🧠 What it does:
- Manages ReplicaSets
- Provides:
  - Rolling updates  
  - Rollbacks  
  - Version control  

---

### 📄 Deployment YAML

```yaml
apiVersion: apps/v1
kind: Deployment

metadata:
  name: nginx-deployment
  annotations:
    kubernetes.io/change-cause: "Initial release"

spec:
  replicas: 3

  selector:
    matchLabels:
      app: nginx

  template:
    metadata:
      labels:
        app: nginx

    spec:
      containers:
        - name: nginx-container
          image: nginx:1.19
```

---

## 🔄 Concept 6: Rolling Updates

> [!important]
> Zero-downtime updates

---

### 🔧 Update Image

```bash
kubectl set image deployment nginx-deployment nginx-container=nginx:1.20
```

---

### 📜 Check History

```bash
kubectl rollout history deployment nginx-deployment
```

---

### 🏁 Output Example
- Revision 1 → nginx:1.19  
- Revision 2 → nginx:1.20  

---

## 🔁 Concept 7: Rollback

> [!important]

### Rollback to Previous

```bash
kubectl rollout undo deployment nginx-deployment
```

---

### Rollback to Specific Version

```bash
kubectl rollout undo deployment nginx-deployment --to-revision=1
```

---

## 🧠 How Deployment Works Internally

> [!info]

- Each update creates:
  - New ReplicaSet
- Old ReplicaSet:
  - Scaled down to 0
- New ReplicaSet:
  - Scaled up

---

### 📊 Example

| Revision | Image        | Replicas |
|----------|-------------|----------|
| RS1      | nginx:1.19  | 0        |
| RS2      | nginx:1.20  | 0        |
| RS3      | nginx:1.21  | 3        |
